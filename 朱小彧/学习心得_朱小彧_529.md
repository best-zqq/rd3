# 多人git协作
.gitignore是忽略文件，它被放置在仓库主目录下，将不需上传的文件的名字写入其中，Git 就会自动忽略它们         
Fork：在别人的仓库中点此按钮会克隆一个完全一样的仓库到你自己的账号中，包括所有分支、提交等，    
但不会克隆 issue，当此仓库发生版本变化，**不会自动同步到你克隆的仓库里**，反之亦然。   
第一作者在仓库里点击collaborate 输入用户名   邀请该人成为合作者，这个人会收到一封邮件，    
点击即可跳转到项目页面，点击fork可copy到自己的仓库里。    

组长可以创建issue  issue在github上会有编号1\2\3\4    
组员完成任务后 commit -m 'fix#1' 即可关闭1号issue     

# PR pullrequest
PR 就是从甲分支向乙分支提一个请求，该请求中有一个或多个提交，    
对方觉得可以、没问题，就合并（merge) 这个请求，    
也就是把请求中所有提交的修改增加到乙分支上， PR 既可以在仓库内，    
也可以跨用户跨仓库    

合并 PR 的方法有三种，分别解释一下：     
- Create a merge commit ：这种方式会在组长仓库的 master 分支上生成一个新的提交，且保留 PR 中的所有提交信息。这是一种常规操作，用得最多。

- Squash and merge ：压缩合并，它会把 PR 中的全部提交压缩成一个。此方法的优点就是让提交列表特别整洁。一个 PR 里有很多提交，每个提交都是很细小的改动，保留这些提交没什么意义，这种情况就使用此方法合并 PR。

- Rebase and merge ：例如 PR 中有 6 个提交，用此方法合并后，组长仓库也会新增 6 个提交。注意，这些提交的版本号与组员的提交不同，此外完全一样。

从 A 向 B 提 PR 后，在 PR 合并或关闭前，A 上所有新增的提交都会出现在 PR 里。   
**作为组员，要时刻保持自己的 master 分支与组长的一致，以避免在下次提 PR 时出现冲突，该操作叫做 “同步主仓库”，组长的仓库就是主仓库。**    
**提 PR、合并 PR 只能在 GitHub 页面上操作。同步主仓库是要用 Git 操作的。**
```
# 首先假设已经从自己的仓库clone了自己的fork下来
 git remote  add  upper  https://....... 添加组长项目地址作为另一个远程仓库
 #git remote -v 查看自己有哪些仓库 默认有一个origin 现在应该有一个upper（名字自己起）
git fetch upper
git rebase upper/master
#git pull --rebase up master 也行
```
使用git pull的会将本地的代码更新至远程仓库里面最新的代码版本（直接改变工作区内容），而使用fetch 则是更新了本地仓库    
# 标签 
创建标签使用 git tag [标签名] -m [备注信息] [提交版本号]     
```
git tag  v1.0  -m '正式版第一版'
```
 git show [标签名] 查看标签详情      
git push origin [标签名]
